# Running Analytics Metadata Repository v1.1

## Purpose

This file is the metadata repository for the Personal Running Intelligence Platform: data catalog, data dictionary, data lineage, data governance, and data lifecycle reference.

It answers:

- What data exists?
- Where does it come from?
- Has it already been parsed?
- Should it enter Excel?
- Where should it land in SQLite later?
- What is the source of truth?
- What is the data granularity?
- How is the data acquired?
- How confident are we in the source and parser?

This metadata repository is the single source of truth for Excel Schema, LDM, SQLite Schema, Parser, Dashboard, and AI Coach design.

## Lifecycle Rule

| Lifecycle | Storage Principle |
|---|---|
| Raw | May enter fact or dimension tables |
| Manual | May enter fact or dimension tables |
| Reference | Controlled reference/master data for dimensions and enums |
| Derived | Prefer view, query, or analysis layer |
| Analysis | Do not write back to core fact tables |

## Priority Rule

| Priority | Meaning |
|---|---|
| P0 | Already available and used |
| P1 | Required for next major version |
| P2 | Valuable, but can be added later |
| P3 | Long-term research |

## Source of Truth Rule

When the same data item appears in multiple systems, `Source of Truth` defines the authoritative source for the platform.

Examples:

| Data Item | Source of Truth |
|---|---|
| Distance | FIT |
| Avg HR | FIT |
| Shoe | Manual |
| Temperature | Weather API |
| Training Purpose | Manual |
| AI Coach Summary | AI Coach |

| Granularity | Example Table |

| Daily | health_daily |

## Acquisition Method Rule

| Acquisition Method | Meaning |
|---|---|
| FIT Native | Direct field from FIT messages |
| FIT Parsed | Parsed or normalized from FIT-native fields |
| FIT Derived | Calculated from FIT-native fields |
| Weather API | Retrieved from weather provider |
| Manual | User-maintained input |
| AI Generated | Generated by AI analysis layer |
| Research | Source not confirmed yet |

## Confidence Rule

| Confidence | Meaning |
|---|---|
| Verified | Confirmed in current system or source design |
| Sample Verified | Verified in sampled FIT/Excel files |
| Partially Verified | Verified only in some files or conditions |
| Assumed | Reasonable design assumption, not fully proven |
| Research | Needs source research |

## Logical Data Type Rule

`Data Type` is the logical platform type, not the SQLite physical type. SQLite schema generation should map these logical types into SQLite storage classes and constraints.

| Data Type | Meaning |
|---|---|
| UUID | Globally unique identifier |
| INTEGER | Whole number |
| DECIMAL | Numeric value with decimals |
| BOOLEAN | True/false value |
| TEXT | Free text or short text |
| ENUM | Controlled vocabulary |
| DATETIME | ISO8601 datetime |
| DATE | ISO8601 date |
| FK | Foreign key reference |
| HASH | Hash / digest value |
| JSON | Structured JSON payload |

## Required / Nullable Rule

| Field | Meaning |
|---|---|
| Required | Whether the importer or application should require the value for the domain workflow |
| Nullable | Whether the final SQLite field may be NULL |

Examples:

| Data Item | Required | Nullable | Reason |
|---|---|---|---|
| Distance | YES | NO | Core activity fact |
| Weather | NO | YES | API may fail or be disabled |
| Stamina | NO | YES | Only available on supported devices/activities |
| Training Purpose | CONDITIONAL | YES | Required once coaching taxonomy is enabled, but may be absent for imported history |
